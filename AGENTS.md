## 4. 認可の影響範囲を閉じる
新しい Middleware / Policy / Gate を追加したら、既存ルートへの巻き込みが無いか一覧で確認する。
管理者機能などの権限グループは専用のルートグループ / プレフィックスに必ず閉じる。
## 5. テスト必須カテゴリ
以下を含む機能は必ず自動テストを先または同時に書く：
- 金額・数量・損益など計算ロジック（算法切替がある場合は各算法ごと）
- 認可（他ユーザーのリソースが漏れない、未ログイン時リダイレクト、権限不足で 403）
- エクスポート（ヘッダ、文字コード、改行、日付フォーマット）
- 外部API連携（モック、タイムアウト、レート制限、リトライ）
テストが無い計算・権限・外部I/O は「完了」と報告しない。
## 6. 完了前の必須チェック
以下が全て通るまで完了報告しない（プロジェクトのコマンド体系に読み替える）：
- 依存再インストール / DBリセット + シード
- 全テスト通過
- フロントビルド成功
- ルート一覧の差分レビュー
テスト件数を減らす / スキップする形の「修正」は禁止。
## 7. クロスプラットフォーム配慮
- OS依存の拡張モジュール（例: pcntl）を必須依存に入れない
- シェルコマンドは PowerShell / bash 両方で通る記法にするか、パッケージスクリプトに寄せる
- パス区切りは forward slash または言語標準の区切り定数を使用
## 8. 完了時レポートのフォーマット
作業終了時に必ず以下のセクションで報告する：
- 実装した機能
- 触った共有ファイル（diff サマリ）
- 追加/変更したスキーマ一覧
- 新規エンドポイント一覧
- 通したテスト件数
- 既知の残課題
- 運用時の注意（対象OS、本番環境、スケジューラ等）
## 9. シークレット・クレデンシャル運用
本プロジェクトは DB 接続情報・API キー等のシークレットを次の原則で扱う：
- `.env` は `.gitignore` 済み。平文パスワードを含むため **絶対にコミットしない**。
- `.env.example` には値を書かない（placeholder のコメントのみ）。ローテーション指針と取得経路をコメントで明示する。
- ローカル開発: `php artisan local:rotate-db-password` でランダムパスワードを都度生成。
  生成後は PostgreSQL 側でも `ALTER USER ... WITH PASSWORD '...'` で同期する。
- 本番: シークレットを `.env` に置かない。ホスティング提供のシークレットマネージャ
  （AWS SSM Parameter Store / GCP Secret Manager / HashiCorp Vault 等）からプロセス環境変数として注入する。
- CI (GitHub Actions): POSTGRES 平文を避けるため、
  スーパーユーザー側は `${{ github.run_id }}` ベース、
  アプリロール側は `openssl rand -hex 32` によるジョブ内ランダム生成を採用する。
- APP_KEY ローテーション時は、セッション / 2FA シークレット / recovery code がすべて無効化されるため、
  事前に全ユーザーへ再セットアップ告知を出すこと。

## 10. AGENTS.md との関係
プロジェクト直下に `AGENTS.md` がある場合：
- 本ルールより `AGENTS.md` の固有規約を優先する
- 競合がある場合は `AGENTS.md` の指示に従う
- `AGENTS.md` が存在しない新規プロジェクトでは本ルールのみを適用する

## 11. 取引所 API 連携の今後の開発方針
- 新しい取引所を追加する前に、公式 API ドキュメントで約定履歴の取得可否、対象商品、過去履歴の取得範囲、レート制限、署名方式、必要権限を確認する。
- 取引所のログイン ID / パスワードは扱わない。ユーザーが発行した API Key / API Secret のみを登録対象とし、売買・出金など更新系権限を要求しない。
- APIキーの登録時は、可能な限り読み取り疎通と権限確認を行う。取引所 API で権限一覧を取得できない場合は、README と個別 docs に運用上の注意を明記する。
- API で取得できない履歴、古い履歴、販売所・簡単売買など対象外履歴は、手動 CRUD または CSV インポートで補完する方針を優先する。
- 外部 API 連携は必ず HTTP モックの自動テストを追加し、署名、ページング、レート制限、リトライ、重複取り込み防止、同期開始日の扱いを検証する。
- 取引所ごとの未対応理由は README の「今後の対応予定・未対応取引所」に反映し、サービス終了や API 利用不可の取引所を実装対象として扱わない。
